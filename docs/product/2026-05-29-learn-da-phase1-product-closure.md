# Learn DA 第一阶段产品闭环改造方案

面向 `Pandas / SQL -> Polars / DuckDB` 迁移学习者

## 1. 项目目标

### 1.1 第一阶段目标

将当前半成品 MVP 收敛为一个可完成完整学习闭环的交互式学习产品，让目标用户能够顺畅完成：

`进入首页 -> 选择学习路径 -> 阅读课程 -> 运行代码 -> 获得 AI 辅助 -> 完成一课 -> 进入下一课`

### 1.2 本阶段不追求

本阶段不追求平台化、账号体系、复杂商业化能力，也不追求课程数量最大化，而是优先做出一个：

- 方向清晰
- 体验完整
- 价值可感知
- 易于继续扩展

的学习型 MVP。

### 1.3 成功标准

第一阶段完成后，用户应该能在 15 到 30 分钟内明显感受到：

- 这个产品是给谁用的
- 我应该先从哪条路径开始
- 我能边学边运行代码
- AI 能结合当前课程和代码帮助我
- 我知道学完这一课后下一步做什么

## 2. 用户画像

### 2.1 主目标用户

有 `Pandas` 或 `SQL` 基础、希望迁移到 `Polars` / `DuckDB` 的学习者。

### 2.2 用户特征

这类用户通常具备以下特点：

- 已经理解数据分析基本概念
- 能读懂基础 Python 或 SQL
- 不需要从零讲“什么是 DataFrame / SQL”
- 更关心“Polars / DuckDB 和我原来用法有什么区别”
- 希望快速上手并形成迁移心智
- 更偏爱“对比 + 可运行示例 + 实战建议”的学习方式

### 2.3 用户核心诉求

用户真正想解决的不是“系统学习一门新语言”，而是：

- Pandas 写法怎么迁到 Polars
- 常见 SQL 分析怎么迁到 DuckDB
- 两者分别适合什么场景
- 如何在真实分析流程中组合使用
- 出错时怎么快速纠正

## 3. 产品定位

一句话定位建议定为：

`一个帮助 Pandas / SQL 使用者快速迁移到 Polars / DuckDB 的交互式学习平台。`

这个定位比“数据分析学习平台”更聚焦，能直接指导：

- 首页文案
- 学习路径设计
- 课程组织方式
- AI 助手提示词方向
- Playground 示例内容

## 4. MVP 边界

### 4.1 本阶段必须完成的能力

第一阶段只聚焦以下 5 个核心能力：

1. 清晰的学习入口
2. 可理解的迁移学习路径
3. 课程与 Playground 强联动
4. Agent 成为课程内辅导员
5. 完成学习后的下一步引导

### 4.2 本阶段暂不做

以下能力统一延后，不进入第一阶段主线：

- 登录 / 注册
- 云端学习进度同步
- 社区讨论
- 成就系统 / 排行榜
- 多模型切换
- 复杂权限体系
- 大规模课程扩充
- 通用聊天机器人化
- 复杂数据持久化分析

## 5. 当前项目现状判断

从现有代码结构看，项目已经具备 MVP 雏形：

- 前端已有首页、学习页、课程详情、Playground、Agent 面板
- 后端已有 learning / playground / agent 三个核心模块
- 已有本地学习状态保存
- 已有课程与示例内容
- 已有代码执行与结果展示能力

但当前问题也比较明显：

### 5.1 主要问题

1. 产品主线还不够聚焦
   首页和页面文案偏“泛平台介绍”，对迁移学习者的价值表达还不够集中。

2. 学习路径虽有分类，但“为什么先学这个”不够明确
   用户能看到内容，但未必知道最合理的学习顺序。

3. Agent 具备能力，但角色感不够强
   更像一个可用的 AI 面板，而不是课程内辅导员。

4. Playground 很强，但与课程闭环还可以更深
   运行代码有了，但“学完 -> 改代码 -> 验证 -> 下一步”链路还不够完整。

5. 完成态设计偏弱
   用户学完一课后缺少明确反馈与下一步引导。

### 5.2 结论

现在最需要的不是新增大功能，而是把已有模块重新组织成一个完整、顺滑、可感知价值的学习闭环。

## 6. 第一阶段产品策略

采用：

`课程驱动型学习闭环`

以课程为主轴，Playground 和 Agent 都围绕课程服务。

### 6.1 为什么选这个策略

原因有三点：

1. 最符合现有结构
   当前项目的课程、Playground、Agent 已经存在，适合围绕课程整合。

2. 改造成本最低
   不需要推翻重来，而是通过信息架构、交互和文案重构提升体验。

3. 最容易做出真正可用的 MVP
   比起做成“AI demo”或“代码实验场”，课程驱动更容易形成清晰价值闭环。

## 7. 页面级改造方案

### 7.1 首页 Home

#### 目标

让用户在几秒内知道：

- 这是给谁的
- 为什么值得学
- 应该从哪里开始

#### 改造方向

1. 首页主标题从“系统学习数据分析实战技能”收敛为更明确的迁移表达，例如：
   - 从 Pandas / SQL，迁移到 Polars / DuckDB
   - 面向已有数据分析基础用户的现代分析工具学习平台

2. 首屏 CTA 只强化两条主路径：
   - 开始 Polars 迁移学习
   - 开始 DuckDB 迁移学习

3. 降低泛展示内容比重，增强“迁移价值说明”：
   - 为什么 Polars 适合 Pandas 用户
   - 为什么 DuckDB 适合 SQL 用户
   - 什么时候组合使用更高效

4. 在首页增加“推荐起步路径”区块，例如：
   - 我会 Pandas -> 先学 Polars Basics
   - 我会 SQL -> 先学 DuckDB Foundations
   - 我两个都会 -> 直接进入组合实战

#### 首页验收标准

- 用户能在 10 秒内理解目标定位
- 用户能在 1 次点击内进入推荐学习路径
- 页面文案整体围绕“迁移学习”而非“泛功能介绍”

### 7.2 学习页 Learning

#### 目标

让用户明确“该学什么”和“先后顺序是什么”。

#### 改造方向

1. 分类逻辑从“内容分类”升级为“迁移路径分类”：
   - Pandas -> Polars
   - SQL -> DuckDB
   - Polars + DuckDB Workflow

2. 每条路径显示：
   - 适合谁
   - 你将获得什么
   - 推荐起点
   - 课程数 / 预计学习时间

3. 在课程卡片中增强迁移语义：
   - 当前课程对应什么旧知识
   - 学完能解决什么问题

4. 筛选和搜索继续保留，但默认视角应优先是“路径浏览”，而不是“内容库检索”

#### 学习页验收标准

- 用户不需要先理解全部分类也能直接进入推荐路径
- 用户能感知每条路径的学习收益
- 用户能快速判断当前课程适不适合自己

### 7.3 课程详情 / 课程型 Playground

#### 目标

把“读内容”和“动手练”真正连成一体。

#### 改造方向

1. 每节课结构统一为：
   - 这节课解决什么迁移问题
   - 核心概念
   - 与 Pandas / SQL 的对应关系
   - 可运行示例
   - 推荐练习动作
   - 学完后建议

2. 强化“加载代码 -> 运行 -> 修改 -> 再运行”的链路
   课程中的代码块不只是展示，而是默认鼓励进入 Playground 实操。

3. 每节课增加轻量“练习任务”，例如：
   - 把筛选条件改成别的列
   - 增加一个 `group by`
   - 用 DuckDB 重写刚才的逻辑

4. 每节课结束后提供明确下一步：
   - 下一课推荐
   - 去自由 Playground 练习
   - 向 AI 请求一题迁移练习

#### 验收标准

- 用户可以不离开当前学习流完成实操
- 每节课至少有一个明确练习动作
- 每节课结尾都有下一步引导

### 7.4 Agent 面板

#### 目标

让 Agent 从“可聊天”升级为“课程内辅导员”。

#### 改造方向

Agent 在第一阶段只强调 4 个核心价值：

1. 解释当前代码
2. 结合当前报错给修复建议
3. 说明当前写法与 Pandas / SQL 的差异
4. 给出当前课程的下一步练习建议

#### 产品策略

- 弱化泛聊天感
- 强化上下文感
- 所有快捷操作都应紧贴当前课程与当前代码

#### 可优先优化的交互

- “解释这段代码”
- “把这段 Pandas 改成 Polars”
- “这个 SQL 场景如何用 DuckDB 写”
- “根据本课给我一个小练习”
- “我下一步该学什么”

#### 验收标准

- 用户能明显感知 AI 是“懂当前页面上下文的”
- Agent 的回答更偏教学辅导，而不是泛泛输出
- 快捷操作能覆盖高频学习卡点

### 7.5 完成反馈与学习延续

#### 目标

解决“学完之后去哪”的问题。

#### 改造方向

1. 课程完成后给出轻反馈：
   - 已完成本课
   - 你掌握了什么
   - 推荐下一课

2. 学习页展示轻量路径进度
   不做复杂成就系统，只做基础完成态和继续学习提示。

3. Playground 保持与课程上下文关联
   让用户回到页面时知道自己上次在练什么。

#### 验收标准

- 用户完成后不会失去方向
- 学习状态有最基本的可见性
- 继续学习入口清晰

## 8. 功能优先级

### P0：必须优先完成

1. 首页定位与 CTA 收敛
2. 学习路径信息架构重构
3. 课程详情与 Playground 联动增强
4. Agent 课程上下文化
5. 每课结束后的下一步引导

### P1：第一阶段内建议完成

1. 课程卡片迁移文案优化
2. 轻量学习进度展示
3. 示例与练习任务标准化
4. 错误提示与空状态体验优化

### P2：可延后

1. 更多高级课程
2. 更强数据预览能力
3. 更复杂的学习记录统计
4. 更丰富的 Agent persona 和记忆

## 9. 里程碑建议

### 里程碑 1：产品定位收敛

目标：
统一首页、学习页、课程页、Agent 的产品叙事。

产出：

- 首页文案改造
- 学习路径命名统一
- 课程结构模板统一
- Agent 快捷动作收敛

### 里程碑 2：学习闭环跑通

目标：
让用户顺畅完成至少 1 条路径中的前几课。

产出：

- 路径入口明确
- 课程到 Playground 跳转顺
- 练习动作明确
- 结课后下一步推荐可用

### 里程碑 3：体验打磨

目标：
让 MVP 从“能跑”提升为“值得继续用”。

产出：

- 错误提示优化
- 空状态优化
- 学习进度可见
- Agent 响应体验优化

## 10. 第一阶段验收标准

项目完成第一阶段后，至少应满足以下验收条件：

1. 用户进入首页后，能明确知道产品目标人群与核心价值。
2. 用户能快速选择一条适合自己的迁移学习路径。
3. 用户能在课程中方便地加载并运行示例代码。
4. 用户在运行出错时，能通过 Agent 获得与当前上下文相关的帮助。
5. 用户完成一课后，能清楚知道下一步该学什么。
6. 产品整体体验围绕“迁移学习闭环”，而不是松散功能集合。

## 11. 建议的下一步执行方式

建议交给 claude code 的顺序：

1. 先做 `首页 + 学习页的信息架构和文案收敛`
2. 再做 `课程详情 / Playground 的学习闭环增强`
3. 再做 `Agent 的课程辅导化改造`
4. 最后做 `完成态、进度和体验收口`
